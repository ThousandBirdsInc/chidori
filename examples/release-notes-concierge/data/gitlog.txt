COMMIT eb7a4f8
DATE 2026-07-02
SUBJECT Performance optimization: cells→locals, fast hasher, inline caches, superinstructions (#88)
BODY
* Profile JS execution end-to-end; land deterministic fast hasher for engine tables

Callgrind (instruction-exact) profiling of the cross-runtime benchmark suite
attributed the interpreter's gap to three structural taxes:

- property-heavy code: 49% of ALL instructions were SipHash + IndexMap
  probing (every o.a get/set hashes the key string with a DoS-hardened
  keyed hash)
- call-heavy code: ~17% allocator traffic (one Rc<RefCell<Value>> per
  binding per call) plus ~8% hashing to resolve globals per call
- tight loops: >52% pure dispatch (step + run_frame)

First fix landed here: replace IndexMap's default RandomState/SipHash with
an in-tree implementation of the Fx hash function (src/fxhash.rs, ~30 lines,
zero new dependencies) for object property maps and the Map/Set/WeakMap/
WeakSet backing stores.

Measured by callgrind (deterministic, contention-immune):
- property_access: 10.98G -> 7.75G instructions (-29.4%)
- fib_recursive:   12.83G -> 12.24G instructions (-4.6%)

Determinism: hashes only decide bucket placement; IndexMap iteration stays
insertion-ordered and lookups are settled by Eq, so the hasher is
unobservable to JS and the replay journal. Fx is unseeded, so bucket layout
is now identical across runs -- strictly more deterministic than
RandomState's per-process random seed. Zero unsafe.

Also adds docs/js-performance-roadmap.md: the measured cross-runtime gaps,
the instruction-level attribution, a re-ranked interpreter roadmap
(property-key atoms, cells->locals, global inline caches, fusion batch,
register-bytecode decision point), and an honest analysis of what a
no-unsafe JIT can and cannot be (native codegen irreducibly requires
unsafe; a Cranelift tier is a product decision with the determinism
constraints already worked out).

Gates: full chidori-js suite (incl. record->replay byte-identity) and the
chidori crate suite green; cross-runtime harness confirms identical RESULT
lines across chidori/node/bun. Test262 gate could not run in this sandbox
(vendoring blocked by network policy) -- relying on CI.

* Rope strings: O(1) concatenation for the += build loop

string_build's profile was 96.5% memcpy: `s += chunk` copied the whole
accumulated string every iteration (O(total^2)). JsString gains a third
representation arm: Rope(Rc<Rope>) -- a lazy concatenation of two
well-formed strings. Each `+` is now one rope node (O(1)); the bytes are
copied exactly once, when first observed byte-wise (as_str), into a
one-shot flat cache. Total byte and UTF-16 lengths are stored on the node,
so `.length` and the engine's MAX_STRING_LEN guard (which now uses a new
non-flattening byte_len(); the old wtf8_bytes().len() would have flattened
on every append and defeated the rope) stay O(1).

Correctness containment:
- Ropes are built only over well-formed (Utf8/Rope) operands; any Wtf8
  operand keeps the eager code-unit path, so surrogate re-pairing at the
  boundary is untouched.
- Below 64 combined bytes concat copies eagerly, keeping short-string
  behavior and allocation profile exactly as before.
- Flatten and Drop are both iterative: a 100k-append chain neither
  recurses on read nor on destruction (Drop dismantles uniquely-owned
  nodes with a worklist).
- Every byte-level observer (wtf8_bytes, code_units, equality, hashing,
  the host JSON boundary) flattens once through as_str; is_well_formed/

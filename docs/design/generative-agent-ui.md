# Generative Agent UI — Agent-Authored, Durable, Replayable Surfaces — Design Doc & Implementation Plan

> **Status:** Proposed (design). Nothing here ships yet. This doc designs a new
> host primitive — **`chidori.ui(surface)`** — that generalizes
> [`chidori.input`](../DESIGN.md) from a plain-text prompt into a rich,
> agent-authored **interactive UI surface**, while preserving the property that
> makes Chidori Chidori: the whole human-in-the-loop session is recorded in the
> call log and **replays deterministically**. The surface description is data the
> agent computes (often *generated by a model*); the human's interaction is the
> only nondeterministic input, and — exactly like `input()`'s answer — it is
> frozen into a `CallRecord`.
>
> **Engine:** the runtime is `chidori-js` (the pure-Rust JS engine); there is no
> other engine path to keep in sync (DESIGN.md, "Current State").
>
> **Related:** [`docs/DESIGN.md`](../DESIGN.md) (host API, `input()`, replay),
> [`docs/signals.md`](../signals.md) (multiplayer push; the inverse primitive),
> [`docs/branching-execution.md`](../branching-execution.md) (fork N futures),
> [`docs/context-management.md`](../context-management.md) (structured
> prompt/output), [`docs/trace-emitter.md`](../trace-emitter.md) (the archived
> viewer-events design — a *render* is a different surface from a *trace*).

---

## 1. Summary (TL;DR)

Add **`chidori.ui(surface, opts?)`**: the agent presents a structured,
agent-authored **surface** (a small, versioned tree of UI components — text,
fields, choices, forms, actions) to a human, pauses, and resolves to a typed
**interaction result** when the human submits. Mechanically it is
[`chidori.input`](../DESIGN.md) with three changes:

1. **The prompt becomes a component tree, not a string.** Today `input()` takes a
   string and at most `{ choices: string[] }` (`sdk/typescript/src/agent.ts`,
   `InputOptions`). A surface is structured data — so the agent can ask for a
   form, a ranked list, a labeled approval with diff, a multi-field record — and
   get back structured JSON instead of one line of text.
2. **The surface can be *generated*.** Because a surface is plain JSON, the agent
   can have a model author it (a `prompt()` with a UI-schema structured output)
   and then present it. That is the "generative" in generative UI: the model
   designs the interface; the runtime makes the interaction durable. The
   generating `prompt()` is itself a recorded host call, so the generated surface
   is reproduced on replay like any other computed value.
3. **A non-blocking display sibling, `chidori.render(surface)`**, streams a
   *display-only* surface update to connected clients (a live "here is what I'm
   doing / here is a draft" panel) the way `log()` streams text — recorded so
   replay reproduces the visual timeline, but never blocking the run.

Delivery is an HTTP `POST /sessions/{id}/ui { interaction }` that resolves a
matching pending pause and resumes, reusing the **exact** `/resume` machinery
(`crates/chidori/src/server.rs`). The recorded `CallRecord` freezes the submitted
interaction, so replay is identical regardless of *which* client rendered the
surface or *how*. ~80% of the machinery already exists (the
`execute_input` pause→persist→resume loop, the SSE streaming worker, the
`PendingHostOperation`/`HostPromiseTable` suspension model); the new work is the
surface schema, the `ui`/`render` host ops, one delivery endpoint, and the
SDK-side renderer.

## 2. Motivation (the "why") — *the human boundary deserves the same treatment as the world boundary*

Chidori's thesis is that **every side effect flows through one recorded boundary**
(README, "the trick is a single boundary"). Effects on *the world* — LLM calls,
tools, HTTP — are richly modeled, captured, and replayable. But the effect on *a
human* is still a single line of text: `chidori.input("Proceed?")` returns a
string. The most valuable agents — triage, review, data entry, approvals,
onboarding, anything genuinely human-in-the-loop — need to *show* something
richer than a prompt and *collect* something more structured than a line:

- **"Approve this action" is not a yes/no question.** A real approval shows a
  diff, the cost, the policy that triggered it, and offers *approve / approve-once
  / edit-and-approve / reject-with-reason*. Today the agent flattens all of that
  into a string and parses a string back.
- **Structured collection beats free text.** "What's the customer's tier, region,
  and renewal date?" is a three-field form, not a prompt the agent has to parse
  and re-ask when the human is ambiguous. A surface returns
  `{ tier, region, renewalDate }` typed.
- **The model can design the interaction.** Given a goal ("collect what you need
  to file this expense"), a model can *generate the form* — the fields, the
  choices, the validation — tailored to the case. That is generative UI, and it
  is exactly the kind of nondeterministic, expensive, hard-to-reproduce behavior
  Chidori exists to tame.
- **A live draft surface is worth more than a log line.** While an agent drafts a
  document or assembles a plan, streaming a *rendered* draft (markdown, a table, a
  candidate diff) to the operator — updated as it goes — is the difference between
  watching a progress bar and watching the work.

**Why Chidori specifically, and why it is more than "send the client some JSON":**
every surface presentation and every human interaction is a recorded host effect
in the durable call log. So a rich, interactive, human-driven session is *still
fully reproducible* — `chidori replay` re-renders the identical surfaces and
re-serves the identical human interactions, with **zero** re-prompting of the
human and **zero** model calls to regenerate the UI. That combination — *a real
interactive UI, yet a deterministic, auditable artifact* — is the differentiator.
A bare websocket-to-React stack gives you the first half and throws away the
second. The call log *is* the record of what the human saw and what they did.

**Relationship to `input()` and to signals.** `input()` is the degenerate case of
`ui()` (a one-field text surface). Signals (`docs/signals.md`) are the *inverse*:
an outside party *pushes* `{name, payload, from}` at an agent-declared listen
point. `ui()` is *agent-initiated request/response with a rich surface* — "I am
showing you this and I will block until you act on it." The three compose (§14):
a surface can be steered by a signal; a multiplayer surface can collect from many
participants.

## 3. Goals / Non-Goals

**Goals**

- An agent can `await chidori.ui(surface)` to present an agent-authored component
  tree, pause, and receive a **typed interaction result** when a human submits.
- A surface is **plain, versioned JSON** — so it can be computed by the agent
  *or generated by a model*, validated by the runtime, sent to any renderer, and
  recorded verbatim in the call log.
- An agent can `chidori.render(surface)` to push a **display-only** surface update
  that streams live to connected clients without pausing the run.
- The whole session **replays deterministically**: a recorded `ui` interaction
  returns the identical result on every replay; a recorded `render` reproduces the
  identical surface in the visual timeline. Neither re-renders nor re-prompts.
- Delivery reuses the existing `input` pause + `HostPromiseTable` + `/resume`
  machinery and the SSE streaming worker — **no parallel suspension subsystem**.
- A **CLI textual fallback** renders surfaces in the terminal for `chidori run`
  (the analogue of `input()`'s `InputMode::Stdin`), so agents that use `ui()` are
  still runnable without a graphical client.
- The reference **TypeScript SDK ships a renderer** (web) and the surface types,
  so app authors render a surface and post an interaction with no bespoke wire
  work.

**Non-Goals (initially)**

- **No general-purpose UI framework.** The surface schema is a small, closed,
  versioned vocabulary of components (§7), not arbitrary HTML/JSX or executable
  client code. Agents describe *intent* ("a select over these options"), not
  layout pixels or scripts. (Security: §16.)
- **No client-side callbacks into the agent.** A surface collects one submission
  per `ui()` call. Mid-surface reactive round-trips (typeahead that calls the
  agent on each keystroke) are out of scope; model them as a sequence of `ui()`
  calls or as signals.
- **No styling/theming engine in v1.** Renderers own presentation; the agent owns
  semantics. A `style` hint field is reserved but unspecified.
- **No durable client state.** The renderer is stateless between surfaces; all
  state lives in the agent and the call log. A reconnecting client re-fetches the
  current pending surface (§11), it does not resync a local store.
- **No new determinism policy knob.** A surface is computed inside the agent under
  the existing policy (fixed clock, seeded randomness); `ui()` records like
  `input()`. No `CHIDORI_TS_*` variant is added.

## 4. Background (verified, with file references)

### 4.1 The pause primitive `ui` mirrors — `execute_input`

`crates/chidori/src/runtime/host_core.rs::execute_input(ctx, args)` is the model
to copy. In order it: pulls the string `prompt` from `args`; `ctx.next_seq()`;
`try_replay_checked(seq, "input")` (replay short-circuit, returns the recorded
result); `replay_completed_host_operation(ctx, seq, "input",
PendingHostOperationKind::Input, args)` (resolve a completed op on resume);
`begin_host_operation_with_function(seq, PendingHostOperationKind::Input,
Some("input"), args)`; `run_host_operation_safepoint(...)` to persist the pending
op; then branches on `ctx.input_mode()`:

- `InputMode::Stdin` reads a line, `resolve_host_operation`, `record_call` with the
  answer, `run_host_operation_completion_safepoint`, returns the string.
- `InputMode::Pause` calls `ctx.set_pending_input(PendingInput { seq, prompt })`
  and returns `Err("{PAUSE_MARKER}: {prompt}")` — the `PAUSE_MARKER` sentinel that
  surfaces the pause up the stack.

`ui()` is structurally identical: the only differences are the *args* (a surface,
not a `prompt` string), the *kind* (`Ui`, not `Input`), and the *Stdin* branch
(render the surface as text and collect a structured answer, §13).

### 4.2 Pending-operation + host-promise substrate

`crates/chidori/src/runtime/snapshot.rs`: `PendingHostOperationKind` (Prompt /
Input / Tool / CallAgent / Http / Timer / Signal / …) is the additive enum `ui`
extends. `PendingHostOperation { id, seq, kind, function, args, created_at }`,
`HostPromiseTable` with `completed_operation(seq, kind, args)` (the match used on
resume), and the `SnapshotManifest` carrying `pending` + `host_promises` are all
reused unchanged. A new `PendingHostOperationKind::Ui` variant deserializes
old manifests fine (serde `snake_case` → `"ui"`).

### 4.3 Persistence + resume delivery (the path `ui` reuses)

A paused run persists `.chidori/runs/{id}/pending.json` and `host_promises.json`.
`crates/chidori/src/server.rs` `resume_session` (the `/sessions/{id}/resume`
handler) resolves the pending op via `complete_persisted_pending_host_operation`,
injects a synthetic `CallRecord` at the pending seq, and re-runs to the next
pause/completion with `run_replay_pausable_with_host_promises_and_vfs`. This is
**exactly** the shape `ui` delivery needs, parameterized by `Ui` + an
`interaction` value instead of `Input` + a string. The signals work already
factored a `complete_pending_and_resume(...)` tail out of this handler
(`docs/signals.md` §9); `ui` calls the same helper.

### 4.4 Streaming + live delivery (what `render` and live `ui` reuse)

`crates/chidori/src/server.rs` has the SSE streaming worker
(`stream_session`, `/sessions/stream`) and the in-memory `ActiveSession`
(`cancelled: AtomicBool`, `cancel_tx`, and — from signals Phase 3 — a live
delivery channel). A pause emits a `paused` SSE event and keeps the stream open;
the worker `select!`s on delivery / cancel / deadline and resolves in-process. A
`render` surface streams as a new SSE event kind on this existing pipe; a live
`ui` pause is delivered in-process exactly like a live signal, no HTTP round-trip.

### 4.5 Tracing — a `ui` call is a span for free

`record_call` streams each call's span by `parent_seq`
(`docs/signals.md` §4.6, `docs/trace-emitter.md`). A `ui` / `render` host call
therefore appears as a span automatically; the surface id, component count, and
chosen action ride in the record and can be stamped as OTEL span attributes.

### 4.6 The CLI / SDK seam

`InputMode::Stdin` vs `InputMode::Pause` (`host_core.rs`) is the existing fork
between "CLI run, answer on the terminal" and "served run, pause to disk." `ui()`
uses the same fork: a CLI run renders the surface textually and reads a structured
answer; a served run pauses and waits for `POST .../ui`. SDK authoring types live
in `sdk/typescript/src/agent.ts` (the `Chidori` interface, `InputOptions`,
`Signal`); the surface types and the renderer are added there.

### 4.7 ⚠️ Prerequisite already paid

The signals work fixed the rust-engine arm of `run_with_context` to surface pauses
via `take_pending_input()` / `take_pending_signal()` → `RunResult{paused}`
(`docs/signals.md` §4.7, shipped). `ui()` adds `take_pending_ui()` to the same
arm; the pause-surfacing plumbing it depends on is already in place.

## 5. Design overview

A **surface** is *agent-authored data*; an **interaction** is *the human's
nondeterministic input*. `ui()` is `input()` with a structured prompt and a
structured answer, and the determinism story is identical: the surface is computed
deterministically inside the agent (so it re-derives on replay), and the
interaction is frozen into the `CallRecord` (so it re-serves on replay). The
renderer is a **live-only** concern — exactly as the signal mailbox is live-only
(`docs/signals.md` §10): a replay run produces the identical result without any
renderer attached.

When the agent calls `chidori.ui(surface)` the runtime, in order: **(1)** replays
a recorded result if this seq is in the journal; **(2)** returns a completed
host-op if one matches `(seq, Ui, {surface})`; **(3)** otherwise — `Stdin` mode:
render textually, collect, record; `Pause` mode: set a `PendingUi`, persist, throw
`PAUSE_MARKER`. Delivery from outside is `POST .../ui { interaction }` that
resolves the pending pause and resumes (reusing `/resume`), or — for a live
streaming run — is delivered in-process. The recorded `CallRecord` freezes the
`interaction`, so replay is identical regardless of how or where it was rendered.

`chidori.render(surface)` is the non-blocking sibling: it records a `render` call
(so the visual timeline replays) and, for a live run, emits an SSE `surface` event
— but it **never pauses** and resolves immediately. It is to `ui()` what `log()`
is to `input()`.

The one non-obvious correctness pillar is **the surface-is-recomputed /
interaction-is-recorded split** (§12): the surface need not be stored to be
reproduced (the agent recomputes it), but it *is* stored anyway (in `args`) so the
trace and any re-render are exact; the interaction is the sole external input and
lives in the result.

## 6. The Surface schema

A surface is a small, **closed, versioned** JSON vocabulary — deliberately not
arbitrary markup. Renderers map each node to native widgets; unknown node `type`s
render as a labeled fallback (forward-compat) rather than failing.

```jsonc
{
  "v": 1,                       // schema version (additive growth only)
  "id": "expense-intake",       // stable id for this surface (agent-chosen)
  "title": "File this expense", // optional header
  "blocks": [                   // ordered display + input nodes
    { "type": "markdown", "text": "I found a **$48.10** receipt from *Acme Cafe*." },
    { "type": "field", "name": "category", "label": "Category", "input": {
        "kind": "select", "options": ["Meals", "Travel", "Supplies"], "default": "Meals" } },
    { "type": "field", "name": "billable", "label": "Billable to client?", "input": {
        "kind": "boolean", "default": true } },
    { "type": "field", "name": "memo", "label": "Memo", "input": {
        "kind": "text", "multiline": true, "required": false } },
    { "type": "table", "columns": ["Item", "Amount"], "rows": [["Latte", "$4.10"], ["…", "…"]] }
  ],
  "actions": [                  // terminal buttons; each maps to a submit `action`
    { "name": "submit", "label": "File it", "primary": true },
    { "name": "skip",   "label": "Not an expense", "style": "danger" }
  ]
}
```

**Block types (v1, intentionally minimal):**

| Type | Purpose | Result contribution |
|---|---|---|
| `markdown` / `text` | Display prose | none |
| `field` | One labeled input (`input.kind`: `text` \| `number` \| `select` \| `multiselect` \| `boolean` \| `date`) | `{ [name]: value }` |
| `group` | Nested labeled set of blocks | nested object under `group.name` |
| `table` | Read-only tabular display | none |
| `image` | Display an image by `url` or VFS path | none |
| `diff` | Before/after text (for approvals) | none |
| `progress` | Determinate/indeterminate status (mostly for `render`) | none |

**The interaction result** the agent receives:

```jsonc
{
  "action": "submit",                 // which `actions[].name` the human chose
  "values": {                         // collected field values, keyed by field name
    "category": "Meals", "billable": true, "memo": "Client lunch"
  }
}
```

Field `input.kind`s carry just enough constraint (`required`, `min`/`max`,
`options`, `pattern`) for a renderer to validate client-side **and** for the
runtime to validate server-side on submission (§16). The schema grows additively;
`v` gates a renderer's understanding and a manifest's compatibility.

The TS SDK exports the schema as discriminated-union types plus a tiny builder so
agent authors get type-checked surfaces without hand-writing JSON:

```ts
import { ui } from "chidori";
const surface = ui.surface("expense-intake")
  .markdown(`I found a **$48.10** receipt from *Acme Cafe*.`)
  .select("category", "Category", ["Meals", "Travel", "Supplies"], { default: "Meals" })
  .boolean("billable", "Billable to client?", { default: true })
  .text("memo", "Memo", { multiline: true, required: false })
  .action("submit", "File it", { primary: true })
  .action("skip", "Not an expense", { style: "danger" })
  .build();
```

## 7. API surface

### 7.1 Agent-facing (`chidori.ui` / `chidori.render`)

```ts
import { chidori } from "chidori";

type Surface = { v: 1; id: string; title?: string; blocks: Block[]; actions: Action[] };
type Interaction<V = JsonObject> = { action: string; values: V };

// Blocking: present an agent-authored surface, pause, and resolve to the
// human's typed interaction. The Chidori analogue of `input()`, but structured.
// With `timeoutMs`, resolves to a `{ action: "__timeout__", values: {} }` sentinel
// (mirrors signals' timeout sentinel; discriminate on `action`).
chidori.ui<V = JsonObject>(surface: Surface, opts?: {
  timeoutMs?: number;
}): Promise<Interaction<V>>;

// Non-blocking: push a display-only surface update to connected clients. Records
// the surface (so the visual timeline replays) and resolves immediately. The
// Chidori analogue of `log()`, but visual. Returns a handle so a later render can
// update the same panel by id.
chidori.render(surface: Surface, opts?: { replace?: boolean }): Promise<void>;
```

Both are durable, promise-returning host operations (DESIGN.md, "Host API"). `ui`
suspends; `render` does not. Neither is on the networking path — surfaces never
touch `fetch`; they ride the host boundary.

### 7.2 Generative surfaces — the model authors the UI

Because a surface is plain JSON, the natural pattern is to have a model produce it
and then present it. This is the generative path; it is *not* a new primitive —
it is `prompt()` (a recorded host call) feeding `ui()` (another recorded host
call):

```ts
// Ask a model to DESIGN the form for this case, constrained to the surface schema.
const surface = await chidori.prompt(
  `Design an intake form to file this receipt. Return only a Surface (schema v1).`,
  { responseSchema: SurfaceSchemaV1, /* structured output, see context-management.md */ }
);
const result = await chidori.ui(validateSurface(surface));   // validateSurface throws on bad shape
```

The generating `prompt()` is journaled, so on replay the *generated surface* is
reproduced from the recorded model response — the UI is as reproducible as any
other model output. `validateSurface` (a pure SDK helper) rejects out-of-schema
surfaces before they reach the host, turning a hallucinated component into an
actionable error rather than a broken render (§16).

### 7.3 SDK types

Add `Surface`, `Block`, `Action`, `Interaction`, the `ui` builder, the
`validateSurface` guard, and `ui` / `render` on the `Chidori` interface in
`sdk/typescript/src/agent.ts` (types + builder; the runtime supplies the host
methods, like `signal`/`input`). Export a framework-agnostic renderer (§13) from a
new `sdk/typescript/src/ui/` module.

## 8. Worked example — *a generative expense-triage agent*

### 8.1 The scenario

An ops agent watches a shared inbox. For each receipt it: (1) extracts the line
items, (2) has a model *design a tailored intake form* for that receipt, (3) shows
the operator the form, (4) on submit, files the expense — pausing cheaply on disk
between each receipt while the operator is away.

### 8.2 The agent (`examples/expense-triage/triage.ts`)

```ts
import { chidori, run, ui } from "chidori";

run(async (batch: { receipts: Receipt[] }) => {
  const filed: Filed[] = [];

  for (const receipt of batch.receipts) {
    const items = await extractLineItems(receipt);          // expensive: vision model

    // Stream a live "working" panel while we think (display-only, never blocks).
    await chidori.render(
      ui.surface("triage-status").markdown(`Reviewing **${receipt.vendor}**…`)
        .table(["Item", "Amount"], items.map(i => [i.name, i.amount])).build(),
      { replace: true },
    );

    // GENERATIVE: let a model design the form for THIS receipt (recorded prompt).
    const designed = await chidori.prompt(
      `Design a v1 Surface to file this receipt. Fields the operator must confirm: ` +
      `category, billability, and a memo. Pre-fill from: ${JSON.stringify(items)}.`,
      { responseSchema: ui.SurfaceSchemaV1 },
    );

    // Present it and block until the operator acts. Run idles on disk meanwhile.
    const r = await chidori.ui<{ category: string; billable: boolean; memo: string }>(
      ui.validate(designed),                                 // throws on out-of-schema
      { timeoutMs: 1000 * 60 * 60 * 24 },                    // a day to respond
    );

    if (r.action === "skip") { await chidori.log("operator skipped", { vendor: receipt.vendor }); continue; }
    if (r.action === "__timeout__") { await chidori.log("triage expired", { vendor: receipt.vendor }); continue; }

    filed.push(await fileExpense(receipt, r.values));        // category/billable/memo, typed
    await chidori.log("filed", { vendor: receipt.vendor, ...r.values });
  }

  return { filed: filed.length };
});
```

Two ergonomics, one boundary:
- `chidori.render(...)` is **non-blocking** — the operator sees a live panel of
  what the agent is doing; the run never waits on it.
- `chidori.ui(...)` **blocks** — the agent has nothing to do until the operator
  acts; it pauses, persists, and idles cheaply on disk for up to a day.

### 8.3 The client (render a surface, post an interaction)

A web client subscribes to the run's SSE stream, renders the pending surface with
the SDK renderer, and posts the interaction:

```ts
import { renderSurface } from "chidori/ui";

// On a `paused` SSE event carrying { kind: "ui", surface }:
renderSurface(surface, container, async (interaction) => {
  await fetch(`${base}/sessions/${runId}/ui`, {
    method: "POST",
    body: JSON.stringify({ interaction }),       // { action, values }
  });
});
```

The same surface renders in the terminal for `chidori run` (§13) — no web client
required to exercise the agent.

### 8.4 Why durability matters here (the payoff)

`chidori trace $RUN` shows, per receipt: the surface that was *generated*, the
surface that was *shown*, and the exact interaction the operator submitted — a
complete audit of what the human saw and chose. And `chidori replay $RUN`
reproduces the **identical** run: the generated forms come back from the recorded
`prompt()` responses, and the operator's submissions come back from the recorded
`ui()` results — **no model calls to regenerate the forms, no human to re-ask.**
A later "why was this filed as Travel?" investigation re-derives the exact form
the operator saw and the exact value they entered. That is the thing a plain
React-over-websocket stack cannot give you: **a real interactive, model-designed
UI session that is also a deterministic, auditable artifact.**

## 9. Host binding & orchestration (rust engine)

### 9.1 chidori-js host binding

In `crates/chidori-js/src/lib.rs::install_chidori_effects`, add methods mirroring
`input` (surface in slot 0, opts JSON in slot 1), forwarding via the existing
`forward_effect` JSON bridge:

```rust
self.vm.define_method(&chidori, "ui", 2, move |vm, _t, args| {
    let surface = args.first().map(|v| vm.value_to_json(v)).unwrap_or(Null);
    let opts    = args.get(1).map(|v| vm.value_to_json(v)).unwrap_or(Null);
    forward_effect(vm, &d, "ui", json!({ "surface": surface, "opts": opts }))
});
// render -> "render" (non-blocking; resolves immediately).
```

**Reused:** `forward_effect`, `define_method`, `Vm::settle` /
`run_jobs_until_blocked`.

### 9.2 New host-op kind & dispatch

- `crates/chidori/src/runtime/snapshot.rs`: add `PendingHostOperationKind::Ui` (one
  additive enum variant; `snake_case` serde → `"ui"`; old manifests still
  deserialize). `render` does **not** need a pending kind — it never suspends — so
  it records a `CallRecord` directly with no `PendingHostOperation`.
- `crates/chidori/src/runtime/rust_engine.rs::dispatch_effect`: add arms
  `"ui" => host_core::execute_ui(ctx, args)` and
  `"render" => host_core::execute_render(ctx, args)`.

### 9.3 `execute_ui` (the core; `crates/chidori/src/runtime/host_core.rs`)

Modeled exactly on `execute_input` (§4.1). The **match key is `args = { surface }`**
(the surface the agent authored) — the interaction is unknown at pause time and
lives in the *result*, mirroring how `input` keys on `{prompt}` and records the
answer in the result. Steps: validate the surface against schema `v` (host-side
guard, §16) → `next_seq()` → `try_replay_checked(seq, "ui")` → completed-host-op
check → then on `input_mode()`:

- `InputMode::Stdin`: render the surface to a textual form (§13), read structured
  answers from stdin, `resolve_host_operation`, `record_call`, return the
  `{action, values}`.
- `InputMode::Pause`: `begin_host_operation_with_function(seq, Ui, "ui", args)`,
  `run_host_operation_safepoint`, `set_pending_ui(PendingUi { seq, surface })`,
  `Err("{PAUSE_MARKER}: ui:{surface.id}")`.

All sub-primitives are reused. `timeoutMs` is recorded on the pause and enforced
server-side, identical to signals' `pending_signal_deadline` mechanism
(`docs/signals.md` §14, Phase 2).

### 9.4 `execute_render` (non-blocking)

`next_seq()` → replay-check (a recorded render replays as a no-op that re-emits the
same SSE event on a live stream, nothing on a headless replay) → `record_call`
with `{ surface }` → if the run is live (`active_sessions`), emit a `surface` SSE
event on the stream → resolve immediately with `null`. No pending op, no safepoint
suspension; it is `log()` with a surface payload and an extra live-stream emit.

### 9.5 Pause / resume state

- `PendingUi { seq, surface }` on the context (sibling of `PendingInput` /
  `PendingSignal`); `set_pending_ui` / `take_pending_ui`.
- `engine.rs::run_with_context` (rust arm) checks `take_pending_ui()` alongside
  `take_pending_input()` / `take_pending_signal()` and returns
  `RunResult{ paused: Some(PendingPause::Ui { seq, surface_id }) }`. The existing
  `RunResult.paused` enum gains a `Ui` arm so the server knows a surface is waiting
  and can hand the surface to clients.

### 9.6 Determinism, nesting, tracing — free from the substrate

A `ui` / `render` call goes through `execute_durable_json_call`-style recording, so
it carries `parent_seq` and streams a span under its parent. The surface id, block
count, and chosen `action` ride in the record; they can be stamped as OTEL span
attributes (`ui.surface_id`, `ui.action`, `ui.block_count`). On parent replay the
recorded result is returned from cache — no re-render, no re-prompt.

## 10. Delivery endpoint & routing (server)

New route in `crates/chidori/src/server.rs` (beside `/resume` and `/signal`):

```
POST /sessions/{id}/ui     body: { interaction: { action, values } }
GET  /sessions/{id}/ui     -> the pending surface (for a (re)connecting client)
```

Routing — the run is in exactly one state:

| Run state | Detection | Action |
|---|---|---|
| **Paused, waiting on a `ui`** | `status == Paused` AND `pending.json` is a `Ui` op | **Validate the interaction against the pending surface** (§16), then **resolve + resume** (reuse `complete_pending_and_resume` with `Ui`; inject synthetic `ui` CallRecord; `run_replay_pausable_with_host_promises_and_vfs`). |
| **Running, live (streaming)** | `id` in `active_sessions` and worker is on a `Ui` pause | **Deliver in-process** (the signals Phase-3 live path): validate, resolve the persisted op, replay-run in the worker — no HTTP round-trip, stream stays open. |
| **Paused on input / approval / signal** | `pending.json` kind ≠ `Ui` | **`409 Conflict`** — a `ui` submission has no queue (unlike signals, a surface is agent-initiated request/response; there is nothing to mailbox). The client should be showing the actual pending surface from `GET .../ui`. |
| **Completed / Failed / Cancelled** | terminal status | `409 Conflict`. |

The `GET .../ui` read lets a freshly connected (or reconnected) client fetch the
current pending surface without waiting for the next SSE frame — it reads
`pending.json`'s `Ui` args. The "resolve + resume" branch is the same
`complete_pending_and_resume(...)` the signals work extracted (`docs/signals.md`
§9), parameterized by `Ui` + the `{action, values}` value.

**Note — no mailbox.** Unlike signals, `ui` does **not** get a pre-arrival inbox.
A surface is agent-initiated: there is no "submission before the agent asked,"
because the client only knows the surface *after* the agent presents it. This is
strictly simpler than the signal delivery path (no `inbox.json`, no
`delivery_seq`, no same-name tie-break).

**Reused:** `complete_persisted_pending_host_operation`, `complete_pending_and_resume`,
`load_persisted_host_promises`, `load_persisted_vfs`,
`run_replay_pausable_with_host_promises_and_vfs`, `SessionStatus`, `session_view`,
the SSE worker. **New:** `ui_session` handler (POST), `pending_ui` reader (GET),
surface + interaction validators, the `surface` SSE event kind.

## 11. Determinism analysis (the crux)

**Claim:** a `ui` interaction recorded in the call log replays identically,
regardless of which client rendered the surface, how it was rendered, or whether
the surface was hand-written or model-generated.

- **The surface is recomputed, not depended-upon-for-replay.** The surface is
  produced by deterministic agent code (under fixed clock / seeded randomness),
  *or* by a `prompt()` whose response is journaled. Either way the pre-`ui()`
  prefix is deterministic ⇒ the agent reaches `ui()` with the **identical**
  surface on every replay. The surface is *also* stored in `args` (so the trace
  and any re-render are exact and a `GET .../ui` is possible), but replay does not
  *need* the stored copy — it recomputes it. This mirrors how `input()`'s `prompt`
  is both recomputed and recorded.
- **The interaction is the sole external input, and it is frozen.** The match key
  is `(seq, PendingHostOperationKind::Ui, args={surface})`. `seq` comes from the
  deterministic `next_seq()` walk → identical on every replay (inductive: the
  deterministic prefix ⇒ same seq). `kind` is structural. So
  `completed_operation(seq, Ui, {surface})` and `try_replay_checked(seq, "ui")`
  find the same `CallRecord`, and the `{action, values}` result is read **verbatim**
  — **no renderer, no client, no human is consulted for an already-recorded seq.**
  Steps (1)/(2) of `execute_ui` short-circuit before any pause or render.
- **`render` is replay-inert.** A recorded `render` has no result the agent
  branches on (it resolves `void`); on a headless replay it does nothing, on a live
  replay it re-emits the identical `surface` SSE event from the recorded surface.
  Either way it cannot change control flow — it is display-only by construction.
- **Generated surfaces are as deterministic as the model call that made them.** A
  surface authored by `prompt()` is reproduced from that prompt's recorded
  response; the subsequent `ui()` records the human's interaction with *that*
  reproduced surface. Replay re-derives the same generated form and re-serves the
  same submission — the generative path adds **zero** new nondeterminism beyond the
  `prompt()` that was already journaled.

The renderer is therefore a **live-only convenience**, the precise analogue of the
signal mailbox (`docs/signals.md` §10): the log is the source of truth; a replay
run can have **no client attached at all** and still reproduce the exact session.

## 12. Rendering targets

A surface is renderer-agnostic data; three renderers ship or are planned:

1. **Web (SDK, v1).** `sdk/typescript/src/ui/renderSurface.ts` — a tiny,
   dependency-free renderer that maps the component tree to DOM, validates fields
   client-side from the schema constraints, and calls back with the
   `{action, values}` interaction to POST. Framework-agnostic (mounts into a
   container element); React/Vue wrappers are thin adapters over it.
2. **CLI textual fallback (v1).** `InputMode::Stdin` renders the surface as a
   labeled textual form: prose blocks print, each `field` prompts in turn
   (`select` shows a numbered menu — reusing the existing `choices` ergonomics),
   `actions` map to a final choice. This keeps every `ui()` agent runnable with
   `chidori run` and unblocks tests without a browser (§19). It is the direct
   generalization of `input()`'s stdin branch.
3. **ACP / editor clients (later).** `crates/chidori/src/acp.rs` already speaks the
   Agent Client Protocol to editor clients. A surface can be carried as an ACP
   content block so an editor renders the agent's form inline. Deferred to a later
   phase; the schema is designed to map cleanly onto ACP's content model.

Unknown block `type`s degrade to a labeled "unsupported block" placeholder rather
than failing the render, so a newer agent (`v: 2` block) stays usable on an older
renderer for the display-only parts.

## 13. Composition with signals and branching

- **Multiplayer surfaces (with signals).** A surface presented to *one* operator is
  `ui()`. To open *one surface to many participants* — e.g. an approval any of
  three on-call engineers may action — present the surface with `render()` (so all
  connected clients see it) and collect the decision with a `signal("approve")`
  whose `from` records *who* acted (`docs/signals.md` §7). `ui()` is the
  single-collector case; `render()` + `signal()` is the multi-collector case. They
  share the surface schema.
- **Compare-and-pick surfaces (with branching).** `chidori.branch(...)`
  (`docs/branching-execution.md`) forks N futures; a natural follow-on is a surface
  that shows the N outcomes side by side and asks the human to pick one. The branch
  results feed a `table`/`group` surface; the `ui()` interaction's `action` selects
  the branch to keep. Branching explores; the surface adjudicates.

Both compositions fall out of the primitives without new mechanism — the surface
is just data that any of `ui` / `render` / `signal` can carry.

## 14. Security & policy

- **Closed schema, no executable client code.** A surface is a *closed* vocabulary
  of declarative components (§6) — no scripts, no raw HTML, no event handlers, no
  arbitrary URLs-as-code. A model-generated surface can therefore only *describe*
  an interface within the vocabulary; it cannot inject behavior. This is the
  central safety property that makes **generative** UI tractable.
- **Two-sided validation.** The host validates the surface against schema `v`
  before presenting (rejecting hallucinated/out-of-schema generated surfaces with
  an actionable error, per DESIGN.md "fail loudly"), and validates the submitted
  interaction against the *pending* surface on delivery: every submitted field
  name must exist in the surface, every value must satisfy its `field` constraints
  (`options`, `required`, `min`/`max`, `pattern`), and `action` must be a declared
  action. A malformed POST is `400`, never delivered to the agent.
- **`image`/`url` blocks are display references, not fetches by the runtime.** The
  renderer loads them under its own origin/CSP; the runtime never dereferences a
  surface URL (it is not on the `fetch` capture path). VFS-path images resolve
  against the snapshot VFS read-only.
- **No new ambient capability.** `ui`/`render` add no `CHIDORI_TS_*` policy knob;
  they ride the existing host boundary and inherit the run's policy. Surfaces carry
  no secrets unless the agent puts them there (the same discipline as `log()`).

## 15. Alternatives considered

- **Keep `input()`; pass JSON in the prompt string, parse JSON back.** Rejected:
  this is what agents do today, and it is exactly the un-typed, un-validated,
  un-renderable status quo this doc removes. No client knows how to render an
  ad-hoc string, and the agent re-parses free text. A typed surface + typed
  interaction is the load-bearing change.
- **Let the agent emit arbitrary HTML / JSX / a client component bundle.** Rejected:
  it breaks the closed-schema safety property (§14) — a model-generated blob of
  HTML/JS is an injection surface and is unrenderable across non-web targets (CLI,
  ACP). A small declarative vocabulary renders everywhere and is safe to generate.
- **A reactive, multi-round-trip surface protocol** (the client calls back into the
  agent on each field change). Rejected for v1: it requires the agent to be live
  for every keystroke, which fights the "pause cheaply on disk" model and
  complicates determinism (each round-trip is a host op). Model multi-step
  interaction as a *sequence* of `ui()` calls — each one a clean durable safepoint.
- **A separate "UI server" outside the runtime** that holds surface state. Rejected:
  it reintroduces the off-log state the whole design avoids — a surface session that
  is *not* in the call log is *not* replayable, which is the entire point. The call
  log is the surface-session-of-record.
- **Reuse signals for the collection side** (`render()` a surface, always collect
  via `signal()`). Considered and kept as the *multiplayer* composition (§13), but
  not as the base primitive: the common case is one agent asking one human one
  thing and blocking on the answer, which is request/response — `input()`'s shape,
  not a push. `ui()` is that shape with a rich surface; signals remain the push.

## 18. Open questions

- **Surface versioning vs manifest.** Should a run's surfaces' max `v` be stamped
  in the snapshot manifest (so a resume on an older binary fails loudly), or is
  per-surface `v` validation at `ui()` time enough? (Lean: per-surface is enough;
  the manifest already pins ABI.)
- **Partial / incremental submission.** Should a long form allow a "save draft"
  action that records a partial interaction and re-presents — modeled as a second
  `ui()` with the prior values pre-filled — or is that purely an agent-side loop?
  (Lean: agent-side loop; no new primitive.)
- **Multiplayer ownership.** For a `render()`+`signal()` multiplayer surface, do we
  want optimistic-lock semantics (first submitter wins, others see "already
  actioned")? That is a signals-layer concern (`from` + tie-break), referenced here
  but decided there.
- **`render` backpressure.** A tight loop of `render()` calls on a slow live client
  — bounded SSE channel with drop-oldest (as `docs/trace-emitter.md` §12 proposed
  for trace events), or coalesce same-`id` renders? (Lean: coalesce by surface id.)
- **Field value types.** Keep `values` as loose JSON validated against the surface,
  or generate a typed result type from the surface at author time (a TS-level
  mapped type over the builder)? (Lean: ship loose-but-validated; add the typed
  builder return as a DX follow-up.)

## 19. Future work

- A typed surface-result inference: the `ui.surface(...)` builder infers the

  fully typed without a manual generic.
- A surface component for **streaming model output** (a `prompt({ type: "draft" })`

- ACP and editor-native rendering (Phase 3), plus a "surfaces" pane in any

  of the generative path.
- Agent-to-agent surfaces: a supervisor agent presents a `ui()` whose "human" is
  another agent that submits via `POST .../ui` — the same primitive, machine on
  both ends, still fully recorded.

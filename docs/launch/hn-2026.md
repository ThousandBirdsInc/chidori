Show HN: Chidori – From-scratch JS engine in Rust for reproducible AI agents

## Body

I posted Chidori here in 2023 [1] as a "declarative framework for AI agents": a reactive graph, an embedded Deno + Starlark interpreter, time-travel debugging, evals. The argument was that agents are long-running and unpredictable, so the only way to make one reliable is to reproduce what it did five steps back. Three years and a near-total rewrite later, that's still the argument. Everything else changed.

The graph was a mistake. People think in if-statements and loops, and there was a deeper problem: agents get generated now, and models write clean TypeScript far better than any bespoke format. A constrained DSL used to look like a safety feature. Once code is what produces your agents, it's a worse compile target. So agents are plain async TypeScript functions. Every side effect (model calls, tools, fetch) goes through the runtime as a recorded host call, and the runtime ends up holding a complete record of the run.

The core of the rewrite: agents don't run on Node or V8. They run on Chidori's own JavaScript engine, built from scratch in Rust. oxc parser, bytecode compiler, stack VM, zero unsafe. It passes 99% of the Test262 tests it executes and skips about 7.5k tests for features it doesn't implement, with every skip listed in the repo. I went this far down because owning the engine is the only way to make replay a hard guarantee: the runtime controls the clock, randomness, and every I/O call the agent can observe. To be precise about the claim, the first run is as non-deterministic as any LLM. What's guaranteed is that re-execution against the recorded journal is byte-identical, and a checkpoint is validated against the agent source it was recorded from, so a stale journal fails loudly.

That exactness is what everything else stands on. Kill the process and resume in a new one. Pause for a human and pick up days later. Fork competing strategies across real OS threads, where each branch writes a disjoint slice of the log, so even parallel runs replay exactly. Commit a recorded run to git as an integration test that costs $0 and runs in milliseconds. And as proof the engine isn't a toy, unmodified React 18 renders on it: an agent can iterate on a component against its own DOM tests, replaying its earlier model calls for free.

I should be honest that a project this size got built by one person directing coding agents: the engine, the Test262 grind, much of the runtime. The reason I could trust code I didn't type is the same loop I'm shipping. Every change gated by conformance tests, every run recorded and replayable. That's also the longer bet: if tools are just TypeScript and agents write good TypeScript, an agent can factor its own hard-won work into typed tools it reaches for later. I'm early on that loop, but it's what this architecture is for.

On the now-crowded field: Temporal replays journaled histories too, and has for a decade, but you split your code into workflows and activities and keep the workflow half deterministic by discipline; break the rule and replay diverges. LangGraph checkpoints between graph nodes. Vercel's eve, from a few weeks ago, records traces of agents that get a real shell. Chidori's difference is that there is no split and no discipline to maintain. Determinism is a property of the engine, so it can't be violated by user code. It also ships as one binary, no cluster, no Postgres, no Node, and chidori run --isolate puts untrusted code in its own child process behind seccomp, Landlock, and an empty network namespace (Seatbelt on macOS).

Honest cost: an interpreter with no JIT runs compute-heavy microbenchmarks 10-40x slower than Node or Bun, worse on tight numeric loops. The benchmark suite is in the repo. Agent workloads are dominated by model and I/O waits where this barely registers, and startup is about 10x faster than Node.

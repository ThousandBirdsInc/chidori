Method & caveats

  • Source. The sticky <!-- chidori-js-benchmarks --> CI comment on each merged PR that touched crates/chidori-js (workflow js-benchmarks.yml): 5 timed runs + 1 warmup per workload, median reported, startup baseline subtracted, results cross-checked across runtimes before timing is trusted.
  • Shared runners. Numbers come from GitHub-hosted runners, so run-to-run noise is real — Node/Bun wobble on identical code gives a sense of the noise floor (e.g. the geomean's dip after #96 is within it). The 10–60× chidori-js movements dwarf it.
  • Near-zero Node times. On closures and property_access V8 nearly eliminates the benchmark loop, and startup subtraction leaves Node at 0.0–1.4 ms. The "chidori ÷ node" column clamps Node at 0.5 ms so those ratios stay finite; read them as "V8 optimized this away", not a precise gap.
  • The suite grew. array_sum appears from #89; string_scan, typed_array, mutual_recursion were added by #100; mixed_helpers by #102 — added precisely because they were expected to be slow, so each debuts at the bottom. The geomean sticks to the nine day-one workloads for comparability.
  • PRs without data. #95 (npm package management) and #99 (safepoint fix, outside chidori-js) didn't trigger the benchmark workflow and are omitted. #93 and #96 are non-perf PRs that still ran the suite — they act as control points. Merge order differs from PR-number order (#94 before #93, #99 before #97).
  • Startup. chidori-js starts in ~2.6–3.1 ms across every snapshot vs ~27 ms for Node and ~10–16 ms for Bun — that advantage predates the sprint and never regressed.
